DESCRIPTION

tabletext(n) is a tool for translating between SQL tables and matching text files. At present it can read a text file, validate its content, and populate SQL tables using the data; at a later time, it may gain the capability to output SQL tables as text files.

tabletext(n) is used as follows:

  • First, an instance of tabletext(n) is created.
  • Next, the table and field methods are used to define the table schemas, and to associate validators with the tables and fields. The chosen table and field names must match SQLite table definitions exactly.
  • The clear method can be used to clear old data from the SQLite database for the tables known to the tabletext(n) instance.
  • The loadfile and load methods can be used to parse input text and populate the SQLite tables.

BASIC SYNTAX

A tabletext(n) input file is a text file which defines the contents of a number of SQL tables. This section describes the file's syntax, which is independent of the chosen schema.

The file may contain comments and table statements. A comment is a line of text beginning with the "#" character: 

# This is a comment.

Table Syntax

A table statement defines the content of a single database table; each table consists of zero or more records. The table statement has this syntax: 

table tableName {
    record
    record
        .
        .
        .
}

For example, 

table employees {
    record
    record
        .
        .
        .
}

Records and Keys

The records of each table are identified by the table's key fields. Each record in the employees table, for example, might be keyed on the employee's badge number. Other tables might have multiple key fields.

Each record is defined by a record statement, which has this syntax: 

table tableName {
    record keyName keyValue ... {
         field
         field
         .
         .
         .
    }
    .
    .
    .
}

For example, 

table employees {
    record badge_number 10034 {
         field
         field
         .
         .
         .
    }

    .
    .
    .
}

Data Fields

In addition to its key field or fields, most tables also have one or more data fields. Data field values are defined using the field statement, which has this syntax: 

table tableName {
    record keyName keyValue ... {
         field fieldName fieldValue
         field fieldName fieldValue
         .
         .
         .
    }
    .
    .
    .
}

Field values containing whitespace must be quoted with double quotes or curly brackets. By convention, double quotes are used for short strings and curly brackets for structured values (e.g., lists) and for text which spills onto more than one line.

For example, 

table employees {
    record badge_number 10034 {
         field name      "John Doe"
         field mail_stop 144-210
         .
         .
         .
    }

    .
    .
    .
}

SCHEMA DEFINITION

The tabletext(n) schema defines the tables that can appear in the input, the fields which can appear in those tables, constraints on the tables and fields, and validation functions to perform additional validation and consistency checks.

Matching the SQL Schema

In general, the tabletext(n) schema is derived from the SQL schema for the tables of interest.

For example, the SQL schema for a table of employee records might look something like this: 

CREATE TABLE employees (
    badge_number INTEGER       -- The employee's badge number
        PRIMARY KEY,
    name         TEXT,         -- The employee's full name
    mail_stop    TEXT,         -- The employee's mail stop
    .
    .
    .
);

The matching tabletext(n) schema must use the same table and field names: 

set tt [tabletext %AUTO%]

$tt table employees
$tt field employees badge_number -key
$tt field employees name
$tt field employees mail_stop

The "$tt" object can now be used to parse employee data from a file with the syntax shown in BASIC SYNTAX. However, little validation is done, beyond ensuring that the badge number is unique for each employee.

Required Fields

There are several options which can be used to constrain field values. For example, the employee's name is a required field, and we'd like it to be unique. We can specify that as follows: 

$tt field employees name -required -unique

Default Values

Some fields may have default values. These are usually entered in the SQL schema; then, a record which does not explicitly specify a value for that field will get the default value.

For example, suppose that most employees share a single mail stop: 

CREATE TABLE employees (
    badge_number INTEGER       -- The employee's badge number
        PRIMARY KEY,
    name         TEXT,         -- The employee's full name
    mail_stop    TEXT          -- The employee's mail stop
        DEFAULT '144-210',
    .
    .
    .
);

Field Validators

You can also specify a field validator, a function which will ensure that the field's value is valid. For example, badge numbers must not only be unique; they must also be five-digit integers, possibly with leading zeros. Field validators are Tcl commands which expect three arguments: the name of the SQLite database object, the name of the current table, and the field value. A validator has two jobs:

  • Throw an error if the value is invalid.
  • Return the value in canonical form.

For example, a badge number validator might look like this: 

proc BadgeValidator {db table value} {
    # FIRST, validate the value and throw an error if it is invalid
    require {[string is integer -strict $value]} \
        "Value is not an integer: \"$value\""

    # NEXT, put it in canonical form
    return [format "%05d" $value]
}

You'd add this validator to the schema as follows: 

$tt field employees badge_number -key \
    -validator BadgeValidator

Table and Record Validation

Similarly, you can define table and record validators as options on the table method, and you can specify that one table depends on one or more other tables. A record validator is a proc that is called after each record is defined; it can check cross-field constraints. A table validator is a proc that is called after the table is defined; it can check cross-record constraints and even insert additional records into the database.

COMMANDS

tabletext name ?options?
Creates a new tabletext(n) object called name. The object is represented as a new Tcl command in the caller's scope; tabletext returns the fully-qualified form of the name.

INSTANCE COMMAND

Each instance of the tabletext object has the following subcommands:

$tt table name ?options...?

Adds a table called name to the schema. The call may include the following validation options:

-tablevalidator cmd

Defines a command to be called after all of the table's records have been defined. If the table does not appear in the input, the validator will be called after all input has been read. The validator can check cross-record constraints, add records to the table, and so forth. If it detects a problem, it should throw an error.

cmd is a command prefix to which two arguments will be added, the name of the SQLite database object, and the name of the table. Any return value is ignored.

-recordvalidator cmd

Defines a command to be called after each record is read from the input and added to the SQL table, to check cross-field constraints. If the command detects a problem, it should throw an error. Any return value is ignored.

cmd is a command prefix to which three arguments will be added, the name of the SQLite database object, the name of the table, and the "ROWID" of the record in the SQL table. The "ROWID" is used to retrieve the record data, i.e., 

proc MyRecordValidator {db table rowid} {
    $db eval "SELECT * FROM $table WHERE ROWID=\$rowid" row {
        # Validate the constraints
        .
        .
        .
    }
}
-dependson list
This option is used to specify that the table depends on one or more other tables; list is a list of their names. If this table appears in the input before any of the listed tables, an error is thrown.
$tt field table field ?options...?

This method defines a new field for a previously defined table. table is the table name, and field is the field name; these must match the names in the SQLite database schema. The options are used to define additional validation constraints.

-key
This field is (a component of) the primary key. Taken together, the values of the fields marked as -key's must be unique for each record.
-unique
Each record must have a unique value for this field. This option is used only for data fields; keys are unique by definition.
-required
Each record in the input must explicitly specify a value for this field; there is no default value.
-validator cmd

Defines a field validator. The validator is called for each "field" statement in the input; see Field Validators for an example.

cmd is a command prefix to which three arguments will be added: the name of the SQLite database object, the table name, and the value to validate. The command must:

  • Throw an error if the value is invalid.
  • Return the value in canonical form (however the application defines that). For example, enum(n) values can be expressed case-insensitively using short names, long names, or index numbers; the database usually contains the short name with the precise case used in the enum definition.
-default
This is the default value for the field. If the text does not specify a field value, this value will be used instead.
$tt clear db
Deletes all records from known tables in SQLite database db. The table schema, at least, must already be defined in db, as tabletext(n) doesn't know the SQL schema. SQLite tables not included in the tabletext(n) schema will not be touched.
$tt get default table field
Returns the -default value for the specified field. The empty string will be returned if -default was not specified with field.
$tt loadfile db dbfile ?preamble?

Reads input text from a file named dbfile, parses it, and loads the data into tables in SQLite database db. The SQLite tables are cleared first.

If the preamble is specified, it contains text that is parsed before the text in the dbfile.

$tt load db dbtext ?preamble?
This method is identical to loadfile, except that the input text is read from the string dbtext, rather than from a file.
$tt validate subcommand ?args? db table value

These methods are for use as field -validator commands. Each one may have arguments which proceed the standard db, table, and value arguments; these are specified by the schema as part of the -validator.

$tt validate vtype vtype db table value
Validates a field given a validation type object. The vtype must have a validate subcommand that takes one argument, the value to validate, and returns the value if valid and throws an error otherwise. Ideally, the validate subcommand should throw error code INVALID.
$tt validate foreign otherTable field db table value
Verifies that the value being validated appears in the field column in some record in the otherTable.
$tt writefile db outfile infile
Writes the contents of tables, records and fields back out to a file specified as outfile. The supplied database object db is used as the source of the data. infile is the name of the file used to initially populate db and is written to outfile's header.
