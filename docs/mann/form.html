DESCRIPTION

The form(n) widget is a ttk_frame(n) configured to display data entry forms: collections of labels, text strings, and field(i) data entry widgets.

The first step in using the form(n) widget is to form register all application specific field(i) widget types. (The field(i) widgets provided by marsgui(n) are registered automatically.

Next, the application using the field create method to create fields in the form. Each field has a symbolic name, a text label, and a registered field type; the caller may also pass widget options directly to the field.

Next, the application asks the form(n) to layout the fields. By default, labels and fields will be layed out in two parallel columns; however, the caller can also specify a layout spec for more complicated layouts.

The fields in the form(n) are referenced by their symbolic names, not by their window names; and they are usually managed as a group. For example, the form(n) has a value of its own, a dictionary of the field values keyed on the symbolic names, and sets of fields can be marked as being contextual, disabled or invalid.

Field Sets

Every field in the form can be in any or all of three sets: invalid, disabled, and context. How a field is displayed and how it behaves depend on the sets it is in.

A field not in any of the sets behaves normally: the user can edit the value.

When a field's value is known to be invalid, the client can move the field to the invalid set. The field is still editable, but the field's label turns red to indicate that there's a problem.

A field can be disabled by the application. A disabled field cannot be edited by the user, and its label is grayed out. The field widget's -state is set to disabled.

The application can also mark a field as a context field. A context field cannot be edited, but the field's label remains black. This set is used for fields that display contextual data for other fields in the form: data that is for the user to see but not to edit.

Standard Field Types

LAYOUT SPECIFICATION

By default, the form is layed out in two parallel columns. The left column contains a field label and the right column the field itself.

For more complex layouts, the caller must specify a layout specification, or "layout spec". The spec is a Tcl-formatted script that contains commands indicating how to lay out each label and field. (The grid(n) geometry manager is used under the covers.)

Layout Commands

The layout commands are as follows:

at r,c field field ?options...?
Places the named field at row r, column c. The options, if given, are passed along to the grid(n) command. For example, one can make the field span two columns using "-columnspan 2".
at r,c label field ?options...?
Places the named field's label at row r, column c, applying any options to the grid(n) command.
at r,c labelfield field ?options...?
Places the named field's label at row r, column c, and the field itself at row r, column c+1, just as though at label and at field were called individually. The options are applyed to the field.
at r,c text string ?options...?
Places the specified text string at row r, column c, and applies the options if any.
at r,c labelframe label spec ?options...?
Places a ttk_labelframe(n) widget at row r, column c. The frame is labeled with the given label, and contains the elements indicated by the spec. The options are applied when gridding the frame into its parent.

Specifying Rows and Columns

Rows and columns are indexed starting at 0, as is usual for the grid(n) geometry manager, and are specified in the layout commands as integers. Thus, to layout a field at 0,0, enter: 

    at 0,0 field myfield

In addition, the codes "=" and "+" can be used, where "=" means "use the same index as before", and "+" means "increment the previous index by one." For example, the standard layout of two parallel columns can be specified as follows: 

    at 0,0 labelfield field1
    at +,= labelfield field2
    at +,= labelfield field3
       .
       .
       .

Note that the at labelfield command increments the column by one automatically; thus, if you wanted two sets of parallel columns you could do this: 

    at 0,0 labelfield field1;  at =,+ labelfield field2
    at +,0 labelfield field3;  at =,+ labelfield field4
       .
       .
       .

COMMANDS

This module defines the following commands.

form pathName ?options...?

Creates a new form widget named pathName, and returns pathName. The widget supports the following options:

-changecmd command
Specifies a command to be called whenever a field's value changes, for any reason whatsoever (including explicit calls to set). A list of the names of the fields whose values actually changed is appended to the command as an argument.
-currentcmd command
Specifies a command to be called whenever a new field receives the focus. The name of the field receiving the focus is appended to the command as an argument.
-state state
From field(i). form(n) widgets have states, just as the normal tk:: and ttk:: widgets do. The valid states are normal and disabled. If the state is disabled then all fields in the form(n) are disabled; if the state normal then fields are enabled or disabled individually.
form register ftype widget ?options...?
Registers a field type with form(n), for use in individual forms. The ftype is a symbolic name for the field type; e.g., the ftype for the enumfield(n) widget is enum. The widget is the fully-qualified name of the widget type, e.g., ::marsgui::enumfield. Any number of widget options and values can be specified as well; this allows one widget type to provide two or more distinct field types.

INSTANCE COMMAND

form(n) widgets have the following subcommands:

pathName cget option
Returns the value of the specified option.
pathName clear
Deletes all fields from the form(n).
pathName configure option value...
Sets the value of one or more of the object's options.
pathName context fields...

Tells the form(n) that the named fields should be marked as context fields (i.e., -state is disabled, but the field label is not grayed out); all others should be marked as normal, non-context fields. The list of fields can be specified as a single argument, or as individual field names on the command line.

If the first element in the list of fields is "+", then the named fields are marked in addition to any that were previously marked. If the first element is "-", then the named fields will no longer be marked, leaving any other marked fields alone.

pathName disabled fields...

Tells the form(n) that the named fields should be disabled (i.e., -state is disabled); all others should be enabled (i.e., -state is normal). The list of fields can be specified as a single argument, or as individual field names on the command line.

If the first element in the list of fields is "+", then the named fields are disabled in addition to any that were previously disabled. If the first element is "-", then the named fields will no longer be disabled, leaving any other disabled fields alone.

When a field is disabled, its label appears grayed out.

pathName field subcommand ?args...?

The subcommands of the field method are used to deal with individual fields in the form(n).

pathName field create field label ftype ?options...?

Creates a field in the form. The field has symbolic name field and text label label, and is of registered field type ftype. The field widget is created with the options specified when it is registered, plus any additional options specified here. Note that the caller should not set the field's -changecmd or -state options, as these are managed by the form(n) itself.

Once all fields are created, the layout method is used to make them appear in the form(n).

pathName field names
Returns a list of the symbolic names of the fields in the order of creation.
pathName field cget field option
Returns the value of named field widget's option.
pathName field configure field option...
Configures the specified option values for the field widget. As noted above, the caller should not set the field's -changecmd or -state options, as these are managed by the form(n) itself.
pathName field ftype field
Returns the named field's field type.
pathName field current
Returns the name of the field that has the input focus.
pathName field win field
Returns the window name of the field's actual widget. This command should be used sparingly.
pathName field get field
Returns the value currently displayed in the named field.
pathName get
Returns the form's current value, a dictionary of symbolic field names and field values. The dictionary will contain the fields in the order in which they were created.
pathName invalid fields...

Tells the form(n) that the named fields contain invalid values; the remainder contain valid values. Marking a field invalid causes the label associated with the field to turn red. The list of fields can be specified as a single argument, or as individual field names on the command line.

If the first element in the list of fields is "+", then the named fields are marked invalid in addition to any that were previously marked invalid. If the first element is "-", then the named fields will no longer be marked invalid, leaving any other invalid fields alone.

pathName layout ?spec?
Lays out out the currently defined fields so that they are visible. See LAYOUT SPECIFICATION for details on layout specs.
pathName set dict
pathName set key value...
Explicitly sets the form's field values given a dictionary of symbolic field names and field values, and calls the -changecmd if at least one field's value has actually changed, i.e., successive calls to set with the same dict will result in at most one call to the -changecmd. It isn't necessary for the dict to contain an entry for every field.

ENVIRONMENT

marsgui(n) requires Tcl 8.5 or later.

To use marsgui(n) in a Tcl script, the environment variable TCLLIBPATH must include the parent of the package directory.
