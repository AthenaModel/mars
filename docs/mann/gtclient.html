DESCRIPTION

gtclient(n) defines an API which allows a commclient(n) to handle game truth published via gtserver(n)/commserver(n). See the EXAMPLE, below, for how to hook up the gtclient(n) to the commclient(n) object.

Once a gtclient(n) object is created and connected to the commclient(n), it will receive all game truth published by the gtserver(n). Other modules in the application can retrieve game truth values at any time using the get method, and can also register a variety of callbacks for handling updates as they occur.

Note that the gtclient(n) API is designed to match the message protocol used by gtserver(n); see the gtserver(n) man page for details of the protocol.

COMMANDS

gtclient name ?options?

Creates a new gtclient(n) object called name. The object is represented as a new Tcl command in the caller's scope; gtclient returns the fully-qualified form of the name.

The gtclient may be created with the following options:

-logger name
The name of the logger(n) object to use to log gtclient(n) activity.
-logcomponent name
The component name to pass to the -logger object when logging messages; defaults to "gt".
-refreshstartcmd command
If defined, the -refreshstartcmd is called in the global scope on startrefresh, before the refresh begins.
-refreshendcmd command
If defined, the -refreshendcmd is called in the global scope on endrefresh, before any watchers are called.
-completecmd command
If defined, the -completecmd is called in the global scope whenever complete method is received.
-db name
Defines the workstation database that stores game truth objects that are received from the gtserver(n).

INSTANCE COMMAND

Each instance of the gtclient object has the following subcommands:

$gtclient cget option
Returns the value of the specified option.
$gtclient configure option value...
Sets the value of one or more of the object's options.
$gtclient startrefresh
Indicates the beginning of a game truth refresh. Watchers will not be called until the refresh is complete.
$gtclient endrefresh
Indicates the end of a game truth refresh. The -refreshendcmd will be called, if defined, followed by all defined watcher callbacks.
$gtclient set name value ?name value...?

Saves the named game truth variables with the specified values, and calls any watcher callbacks (see watch, below).

Note that if this method is called during a refresh, the watchers will not be called until the refresh is over.

$gtclient unset name ?name...?
Unsets the named game truth variables. The variables are erased from the gtclient(n)'s memory.
$gtclient clear
Unsets all game truth variables.
$gtclient complete
Notifies the client that the current set of game truth variables is consistent and complete; if defined, the -completecmd is called.
$gtclient get name
Returns the value of the named game truth variable.
$gtclient var name
Returns the fully-qualified variable name associated with the named game truth variable, for use as a -textvariable.
$gtclient watch name command

Defines a watcher callback on the named game truth variable; whenever the variable's value is updated, the new value will be append to command and the result evaluated in the global scope. All defined watchers are also called after each refresh, immediately after the -refreshendcmd.

To delete a watcher, set it to the empty string.

$gtclient class class table idcolumn
Defines a class of simulation object that will be created, updated and perhaps deleted. Each class has a corresponding table in the workstation database. This table is the one that has rows created, updated or deleted. The idcolumn specifies which column in the table is the key column.
$gtclient update class id ?dict?
Receives a simulation object with the supplied class from a gtserver(n). If an object with the supplied id already exists, the workstation database is updated with the information in the dict. Otherwise the workstation database has a new record created that has the supplied id and data in the supplied dict. A call is made to the callback supplied in the onupdate command, if there is one. The callback has appended to it "update $id $dict".
$gtclient delete class id
Receives a delete message from the gtserver(n) with the supplied class and id. A call is made to the callback supplie in the onupdate command, if there is one. The callback has appended to it "delete $id $dict", where dict is queried from the workstation database. . The item is then removed from the workstation database.
$gtclient onupdate class prefix

Defines a callback that should be made whenever the client receives a create, update or delete message from the gtserver(n). The callback has appended to it the string "create", "update" or "delete" along with the id and dict for the object.

To delete an onpudate command set it to the empty string.

EXAMPLE

The gtclient(n) is initialized as follows. Note that if the application has a single gtclient(n), it is conventionally called ::gt, as shown; however, any name will do, provided that the commclient(n)'s alias is "gt". 

# FIRST, create the gtclient(n) object
gtclient ::gt \
    -logger ::log \
    -refreshendcmd [list puts "Got a refresh!"] \
    -db ::wsdb

# NEXT, initialize some variables to known values, so that the labels
# they'll be attached to will show something.
gt set zulu "??????????????"

# NEXT, define a class that can be created, updated or deleted
gt class foo fooTable id

# NEXT, define a callback for this class of object
gt onupdate foo [list UpdatedFoo]

# NEXT, insert an object into the workstation database
gt create foo 9 {bar 1 name "Fred" id 9}

# NEXT, update an existing object
gt update foo 9 {name "Wilma"}

# NEXT, delete an object from the database
gt delete foo 9

# NEXT, define watchers
gt watch somevar [list SomevarWatcher]

# NEXT, create the commclient(n) object
commclient ::cc \
   -logger ::log \
   -logcomponent cc \
   -server ...

# NEXT, hook the gtclient(n) to the commclient(n):
cc alias gt ::gt
